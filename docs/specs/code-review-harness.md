# 製品仕様書: コードレビューハーネス（Evaluator 静的レビューモード拡張）

## 概要

千手院ウェブサイトの変更コードを、既存の Evaluator サブエージェントの新モード「静的コードレビューモード」として静的レビューし、ターミナル（Claude Code のチャット）に結果を直接表示する仕組みを整備する。従来の Anthropic Message API 直呼び出し版 CLI（`npm run review` / `scripts/` 配下）をサブスクリプション内で完結する Evaluator の静的レビューモードに置き換える。新規サブエージェントは作成せず、既存の Evaluator を拡張する形で実現する。

## ターゲットユーザー

- 千手院ウェブサイトの開発者（本人）
- コミット前・プッシュ前にセルフレビューを行いたい人
- Generator が実装した差分を、Evaluator のブラウザ検証とは別軸で静的レビューしたい人
- Claude Code のサブスクリプションの範囲で完結させたい（Message API の従量課金を避けたい）人

## 機能一覧

### 機能 1: Evaluator への静的コードレビューモード追加

- **説明**: 既存の Evaluator サブエージェント定義（`.claude/agents/evaluator.md`）に「静的コードレビューモード」のセクションを追加する。Evaluator が 2 つのモードを持つ設計にする。
  - **ブラウザ検証モード**（既存）: Playwright で UI 検証、閾値 3/5 で合否判定、不合格ならパイプラインをブロック
  - **静的レビューモード**（新規）: コード diff の静的レビュー、参考情報のみ、ブロックしない
- **ユーザーストーリー**: 開発者として、コード品質の検証もブラウザ動作の検証も「Evaluator」という 1 つのエージェントに一本化したい。なぜなら、個人開発でサブエージェントが増えすぎるとオーバーヘッドになり、「コードを検証する人 = Evaluator」という直感的な構造にしたいからだ。
- **受け入れ基準**:
  - [ ] `.claude/agents/evaluator.md` に「ブラウザ検証モード」と「静的レビューモード」の 2 モードが明記されている
  - [ ] 各モードの役割・呼び出し条件・判定方式の違いが明確に記述されている
  - [ ] 静的レビューモードのレビュー観点が 4 つ（コード品質 / プロジェクト規約遵守 / セキュリティ・アクセシビリティ / パフォーマンス）に整理されている
  - [ ] severity が critical / warning / info の 3 段階で定義されている
  - [ ] レビュアー像として「シニアエンジニア。過剰指摘を避け、本当に修正価値のある指摘のみを行う」ことが明記されている
  - [ ] 対象拡張子（`.ts` `.tsx` `.css` `.json` `.mjs` `.js`）がルールとして明示されている
  - [ ] 静的レビューモードではブラウザ操作・UI 検証を行わないことが明記されている
  - [ ] ブラウザ検証モードの既存内容（Playwright MCP による検証、評価基準と閾値、レポート出力先など）が損なわれていない
  - [ ] 既存コードベース（Next.js 16 / React 19 / Tailwind CSS 4 / TypeScript strict）の規約に言及している
- **優先度**: 高

### 機能 2: 起動インターフェースとレビュー対象の決定

- **説明**: 静的レビューモードを、スラッシュコマンド（`/review`）および自然言語（「レビューして」「コードレビューお願い」等）のいずれからでも呼び出せるようにする。既定では `git diff HEAD`（未コミット変更）をレビュー対象とし、引数で特定ファイルまたはパスを指定した場合はそのファイル／パスを対象にする。対象拡張子以外はスキップする。
- **ユーザーストーリー**: 開発者として、スラッシュコマンドでも普段の会話の流れでも静的レビューを呼び出したい。また、特に指定しないときは「今ワーキングツリーに積んでいる変更」を、必要なときだけ個別ファイルやディレクトリをレビューしてほしい。なぜなら、状況に応じて自然に使い分けたいからだ。
- **受け入れ基準**:
  - [ ] スラッシュコマンド `/review` から静的レビューモードが起動できる
  - [ ] 自然言語（日本語の「レビューして」「コードレビューお願い」等）から静的レビューモードが起動できる
  - [ ] どちらの入り口から起動しても、同一のレビュー観点・severity 区分・出力フォーマットが適用される
  - [ ] 引数なしで呼び出した場合、`git diff HEAD` の変更ファイルのみがレビュー対象になる
  - [ ] 引数で単一ファイルを指定した場合、そのファイルがレビュー対象になる
  - [ ] 引数でディレクトリ／glob 風のパスを指定した場合、配下の対象拡張子ファイルがレビュー対象になる
  - [ ] 対象拡張子以外のファイルは、対象ファイル選定の段階で自動的に除外される
  - [ ] 対象ファイルが 0 件の場合、その旨をチャットに明示して終了する
  - [ ] 呼び出し方の例が Evaluator 定義またはプロジェクトドキュメントに明記されている
- **優先度**: 高

### 機能 3: レビュー観点・severity・出力フォーマットの定義

- **説明**: レビュー観点（4 つ）、severity（3 段階）の基準を定義し、レビュー結果のターミナル表示フォーマットを整備する。結果はファイル保存せず、Claude Code のチャットに直接表示する。静的レビューモードは参考情報のみであり、合格/不合格の判定やパイプラインのブロックは行わない。
- **ユーザーストーリー**: 開発者として、指摘の「重要度」と「カテゴリ」が一目で分かるレビューをその場で受けたい。かつ、レビューはあくまで参考にしたい。なぜなら、critical のみ即対応・info は参考と判断でき、レポートファイルの管理も不要で、最終的な採否判断は人間が行いたいからだ。
- **受け入れ基準**:
  - [ ] 4 つのレビュー観点それぞれについて、何を見るかが具体例付きで説明されている
  - [ ] 各 severity（critical / warning / info）の基準が定義されている
  - [ ] 「些末な指摘は控える」「指摘は本当に修正価値のあるもののみ」というレビュアー方針が明記されている
  - [ ] 1 ファイルあたりの指摘件数や、同一問題の重複指摘に関するガイドラインがある
  - [ ] プロジェクト規約遵守の観点で、CLAUDE.md のデザインシステム（カラー、フォント、ユーティリティクラス）や命名規則との整合をチェック対象としている
  - [ ] レビュー結果がチャットに直接表示される
  - [ ] 結果ファイル（`docs/reports/` 等）への保存は行わない
  - [ ] ファイルごとに、ファイルパス・行番号（特定できる場合）・severity・カテゴリ・指摘内容が示される
  - [ ] 指摘が 0 件のファイルは簡潔に「問題なし」と分かる扱いにする
  - [ ] 最後に全体サマリー（対象ファイル数、severity 別件数）が示される
  - [ ] 静的レビューモードは合格/不合格の判定を出さない
  - [ ] 静的レビューモードの結果によって他のエージェントやパイプラインがブロックされないことが Evaluator 定義に明記されている
  - [ ] ブラウザ検証モードの閾値制度（3/5 判定）との役割分担が明確に記載されている
- **優先度**: 高

### 機能 4: Message API 版 CLI の撤去と運用ドキュメント更新

- **説明**: 静的レビューモードへ移行後、`scripts/` 配下の Message API 直接呼び出し版コードレビュー CLI と関連設定（`npm run review` エントリ、関連依存関係や環境変数ドキュメント）を撤去する。CLAUDE.md の「コードレビュー CLI」節と「サブエージェント運用ルール」節を、Evaluator の 2 モード体制（ブラウザ検証 + 静的レビュー）を反映した内容に更新する。
- **ユーザーストーリー**: 開発者として、新方式への移行後は古い CLI を完全に退役させ、ドキュメントも一本化したい。なぜなら、二重運用による混乱・従量課金のリスク・ドキュメントの食い違いを避けたいからだ。
- **受け入れ基準**:
  - [ ] Message API 直接呼び出し版 CLI（`scripts/` 配下）がリポジトリから削除されている
  - [ ] `npm run review` のスクリプトエントリが `package.json` から削除されている
  - [ ] Reviewer で不要になった依存関係・環境変数の案内が整理されている（実際に不要な依存関係がある場合）
  - [ ] CLAUDE.md の「コードレビュー CLI」節が Evaluator 静的レビューモードの案内に更新されている
  - [ ] CLAUDE.md の「サブエージェント運用ルール」節が、Evaluator の 2 モード（ブラウザ検証 / 静的レビュー）を前提に更新されている（役割分担表、Evaluator 固有ルールの更新を含む）
  - [ ] 削除・更新により `npm run build` と `npm run lint` が引き続き成功する
- **優先度**: 中

## スプリント計画

### スプリント 1: Evaluator 静的レビューモードの定義と起動インターフェース
- 機能: 機能 1（Evaluator への静的コードレビューモード追加）、機能 2（起動インターフェースとレビュー対象の決定）
- ゴール: `.claude/agents/evaluator.md` にブラウザ検証モードと静的レビューモードの 2 モードが定義され、スラッシュコマンド `/review` と自然言語の両方から静的レビューモードを呼び出せる状態。レビュー対象の決定ロジック（`git diff HEAD` 既定 + 引数指定）も定義に含まれる。既存のブラウザ検証モードの内容は損なわれていない。

### スプリント 2: レビュー観点・severity・出力フォーマット・判定ポリシーの整備
- 機能: 機能 3（レビュー観点・severity・出力フォーマットの定義）
- ゴール: 4 観点と 3 段階 severity の基準、過剰指摘抑止の方針、ターミナル表示フォーマット、およびブロックしない判定ポリシーが Evaluator 定義に組み込まれ、静的レビューモードの仕様が完成した状態。

### スプリント 3: Message API 版 CLI の撤去とドキュメント更新
- 機能: 機能 4（Message API 版 CLI の撤去と運用ドキュメント更新）
- ゴール: 旧 CLI（`scripts/` 配下）が削除され、CLAUDE.md が Evaluator の 2 モード体制に更新され、`npm run build` と `npm run lint` がエラーなしで通る状態。

## 成功指標

- Evaluator サブエージェントの定義に「ブラウザ検証モード」と「静的レビューモード」の 2 モードが明確に記述されている
- 静的レビューモードがスラッシュコマンド `/review` と自然言語の両方から呼び出せる
- 引数なしの呼び出しで `git diff HEAD` に含まれる対象拡張子のファイルすべてにレビューが走り、ターミナルに結果が表示される
- 引数で指定した特定ファイル／パスに対してもレビューが走り、結果が表示される
- レビュー結果に critical / warning / info の区別があり、4 つの観点に沿った指摘が含まれる
- 静的レビューモードの結果はファイル保存されず、チャットにのみ表示される
- 静的レビューモードは合格/不合格を判定せず、パイプラインをブロックしない
- ブラウザ検証モードの既存機能（Playwright MCP 検証、閾値判定、レポート出力）が損なわれていない
- `scripts/` 配下の Message API 版 CLI と `npm run review` が撤去されても、`npm run build` と `npm run lint` がエラーなしで通る
- CLAUDE.md が Evaluator の 2 モード体制を反映した内容に更新されている

## 制約・前提条件

- Claude Code のサブスクリプションに含まれるサブエージェント機能の範囲内で完結させる（Anthropic Message API を従量課金で呼び出さない）
- 新規サブエージェントは作成しない。既存の Evaluator（`.claude/agents/evaluator.md`）を拡張する
- 静的レビューモードはブラウザ操作や UI 検証を行わない（UI 検証はブラウザ検証モードの責務）
- レビュー対象の拡張子は `.ts` `.tsx` `.css` `.json` `.mjs` `.js` のみ
- severity は critical / warning / info の 3 段階のみ使用する
- レビュアー像はシニアエンジニア。過剰指摘を避け、本当に修正価値のある指摘のみを行う
- 静的レビューモードの結果は `docs/reports/` などへのファイル出力を行わない（チャット表示のみ）
- 静的レビューモードは Evaluator のブラウザ検証モードの閾値制度（3/5 判定）を持たず、パイプラインをブロックしない参考情報として位置付ける
- 既存コードベース（Next.js 16 / React 19 / TypeScript 5 strict / Tailwind CSS 4 / GitHub Pages 向け静的エクスポート）の構造と規約を前提とする
- 仕様書が「何を作るか」に集中する方針に則り、具体的な実装方法・ファイル構成・プロンプト文言は本書では指定しない
- 既存の CLAUDE.md における「サブエージェント運用ルール」節の共通ルール（仕様書が唯一の信頼源、ドキュメント駆動、スプリント単位で進行、ビルド通過必須、既存コードを尊重）に従う
