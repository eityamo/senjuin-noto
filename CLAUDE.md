# CLAUDE.md — 千手院ウェブサイト

## プロジェクト概要

石川県穴水町の真言宗寺院「海臨山 千手院」のウェブサイト。能登半島地震からの復興ストーリーを軸とした寺院紹介サイト。

## 技術スタック

- Next.js 16（静的エクスポート） + React 19 + TypeScript 5（strict）
- Tailwind CSS 4（インラインテーマ設定）
- GitHub Pages にデプロイ（`/out` ディレクトリ）
- パスエイリアス: `@/*` → `./src/*`

## よく使うコマンド

- `npm run dev` — 開発サーバー起動（localhost:3000）
- `npm run build` — 静的ビルド（`/out` 生成）
- `npm run lint` — ESLint チェック

## ディレクトリ構造

```
src/
├── app/
│   ├── layout.tsx       # ルートレイアウト（メタデータ、フォント）

│   └── globals.css      # グローバルCSS（Tailwindテーマ）
```

## サブエージェント運用ルール

### 概要

3つのサブエージェントが **Planner → Generator → Evaluator** のパイプラインで協働する。

| エージェント | 役割 | 入力 | 出力 |
|-------------|------|------|------|
| Planner | 要望を仕様書に変換 | ユーザーのプロンプト | `docs/specs/{name}.md` |
| Generator | スプリント単位で実装 | 仕様書 + スプリント番号 | コード + `docs/reports/sprint-{N}-self-eval.md` |
| Evaluator（ブラウザ検証モード） | 実装をブラウザで検証 | 自己評価レポート + 仕様書 | `docs/reports/sprint-{N}-evaluation.md` |
| Evaluator（静的レビューモード） | コード差分を静的レビュー | `git diff HEAD` または指定ファイル | チャットに直接表示（ファイル保存なし） |

### ファイル規約

すべてのドキュメントは以下のパスに配置する：

- 仕様書: `docs/specs/{project-name}.md`
- 自己評価レポート: `docs/reports/sprint-{N}-self-eval.md`
- 評価レポート: `docs/reports/sprint-{N}-evaluation.md`

### パイプラインフロー

```
1. ユーザーがプロンプトを入力
2. Planner → 仕様書を生成 → docs/specs/ に保存
3. ユーザーが仕様書をレビュー・承認
4. Generator → スプリント1を実装 → 自己評価レポートを保存
5. Evaluator → 実装を検証 → 評価レポートを保存
6. 合格 → 次のスプリントへ（4に戻る）
   不合格 → Generatorが修正（4に戻る）
7. 全スプリント完了 → 終了
```

### エージェント間のルール

#### 共通ルール

1. **仕様書が唯一の信頼源** — 実装・検証はすべて `docs/specs/` の仕様書に基づく。口頭指示や推測で動かない
2. **ドキュメント駆動** — 各エージェントは必ずレポートを `docs/reports/` に保存してから次工程に引き渡す
3. **スプリント単位で進行** — 1スプリント = 1機能。複数機能の同時進行は禁止
4. **ビルド通過必須** — `npm run build` と `npm run lint` がエラーなしで通ることは全工程の前提条件
5. **既存コードを尊重** — コードスタイル、ディレクトリ構造、命名規則は既存コードに合わせる

#### Planner 固有ルール

- 技術選択・実装方法を指定しない（「何を作るか」のみ）
- 必ず既存コードベースを確認してから仕様書を作成する
- 1スプリント = 1機能に分割し、優先順位を明確にする
- 出力先: `docs/specs/`

#### Generator 固有ルール

- 仕様書に書かれていないことは実装しない
- 実装後は必ず `npm run build` で確認する
- 自己評価レポートには問題点を正直に記載する
- 出力先: コード変更 + `docs/reports/sprint-{N}-self-eval.md`

#### Evaluator 固有ルール

Evaluator は **ブラウザ検証モード** と **静的レビューモード** の2つのモードを持つ。

**ブラウザ検証モード**（パイプライン内の品質ゲート）:

- **必ずブラウザで実際に操作して確認する**（Playwright MCP を使用）
- 検証前に `npm run dev` でサーバーを起動し、`http://localhost:3000` にアクセスする
- レスポンシブは3サイズ（375x667, 768x1024, 1440x900）で確認する
- 閾値（3/5）を1つでも下回れば不合格。例外なし
- バグには再現手順を必ず付ける
- 出力先: `docs/reports/sprint-{N}-evaluation.md`

**静的レビューモード**（参考情報、パイプラインをブロックしない）:

- `/review` コマンドまたは自然言語（「レビューして」等）で呼び出す
- 対象: `git diff HEAD`（既定）または引数でファイル/ディレクトリを指定
- レビュー結果はチャットに直接表示する（ファイル保存なし）
- 合格/不合格の判定は行わない（参考情報のみ）
- ブラウザ操作・UI検証は行わない
- Message API 不使用、サブスクリプション内で完結

### 不合格時のフロー

1. Evaluator が不合格判定 + 具体的な修正指示をレポートに記載
2. Generator が評価レポートを読み、指摘箇所を修正
3. Generator が修正版の自己評価レポートを保存（`sprint-{N}-self-eval-v2.md`）
4. Evaluator が再検証（同じ基準で再評価）
5. **最大3回の修正ループ**。3回不合格の場合はユーザーに判断を仰ぐ

### オーケストレーション（メインエージェントの役割）

メインエージェント（Claude Code 本体）がパイプラインを管理する：

1. ユーザーの要望を受け取り、適切なサブエージェントを呼び出す
2. 各サブエージェントの出力ファイルパスを次のエージェントに渡す
3. 合格/不合格の判定に基づき、次のアクションを決定する
4. ユーザーへの進捗報告を行う

## 静的コードレビュー（Evaluator 静的レビューモード）

Evaluator サブエージェントの「静的レビューモード」で、コミット前のセルフレビューを行える。Anthropic Message API の従量課金は不要で、Claude Code サブスクリプション内で完結する。

### 使い方

- `/review` コマンド — `git diff HEAD` の変更をレビュー
- `/review src/app/page.tsx` — 指定ファイルをレビュー
- `/review src/app/` — 指定ディレクトリ配下をレビュー
- 自然言語（「レビューして」「コードレビューお願い」等）でも呼び出し可能

### 特徴

- 対象: `git diff HEAD`（既定）または引数でファイル/ディレクトリを指定
- 出力: チャットに直接表示（ファイル保存なし）
- 判定: 合格/不合格の判定は行わない（参考情報のみ）
- severity: `critical` / `warning` / `info` の3段階
